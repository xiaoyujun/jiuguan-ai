# 前端页面优化计划

## 1. 结论

当前项目的前端页面确实存在比较明显的重复建设，主要不是“完全一模一样的整页复制”，而是以下几类问题同时存在：

1. 模板层没有统一基模板，公共头部、资源引入、返回按钮、基础容器结构在多个页面重复维护。
2. 同类页面各自维护一套搜索、列表、弹窗、表单、按钮、空状态、加载状态，复用度偏低。
3. 存在高相似度的成对页面与成对样式文件，已经具备抽公共组件的条件。
4. 存在未引用的旧版脚本和缺失的静态资源引用，说明前端资产已经开始出现分叉和沉积。
5. 少数核心页面体量过大，说明组件边界不清，后续维护成本会继续上升。

这个项目适合做“渐进式前端收敛”，不建议一次性重构全部页面。

## 2. 当前重复现象证据

### 2.1 模板层

已检查 `web/templates/` 下 28 个 HTML 模板，结果如下：

1. 所有模板都是完整 HTML 页面。
2. 没有任何模板使用 Jinja `extends`。
3. 这意味着 `<head>`、`meta`、通用样式、通用脚本、页面壳结构目前基本靠复制维护。

直接影响：

1. 新增一个公共资源时，需要改很多页面。
2. 通用导航和安全区适配容易出现页面间不一致。
3. 样式和脚本加载策略难统一，缓存参数策略也不一致。

### 2.2 明显重复的页面对

#### A. 角色切换弹窗 vs 玩家切换弹窗

涉及文件：

1. `web/templates/role_switcher_modal.html`
2. `web/templates/player_switcher_modal.html`
3. `web/static/js/role_switcher_modal.js`
4. `web/static/js/player_switcher_modal.js`
5. `web/static/css/role_switcher_modal.css`
6. `web/static/css/player_switcher_modal.css`

观察结果：

1. 两个模板结构高度一致，相似度约 `0.919`。
2. 两个 CSS 文件相似度约 `0.761`。
3. 两个 JS 文件相似度约 `0.547`。
4. 差异主要集中在文案、接口对象名称、字段命名。

判断：

这组页面已经不是“风格相近”，而是典型的同一组件两份实现，优先级很高。

#### B. 世界书管理相关页面

涉及文件：

1. `web/templates/storybook.html`
2. `web/templates/keyword_worldbook.html`
3. `web/templates/global_worldbook.html`
4. `web/static/js/storybook.js`
5. `web/static/js/keyword_worldbook.js`
6. `web/static/js/global_worldbook.js`
7. `web/static/css/storybook.css`
8. `web/static/css/keyword_worldbook.css`
9. `web/static/css/global_worldbook.css`

观察结果：

1. 三个页面都在做“列表 + 搜索/筛选 + 编辑弹窗/面板 + 批量操作”这一类管理界面。
2. 页面容器、头部、列表区、空状态、编辑区的组织方式很接近。
3. 区别主要在业务字段和操作按钮，不在页面骨架。

判断：

适合抽成统一的“管理页壳 + 列表工具栏 + 编辑弹窗”模式，而不是继续各写一套。

#### C. 管理类页面的结构重复

涉及代表文件：

1. `web/templates/character_management.html`
2. `web/templates/player_management.html`
3. `web/templates/model_config.html`
4. `web/templates/event_manager.html`
5. `web/templates/free_event.html`

观察结果：

这些页面普遍重复以下结构：

1. 页面标题区
2. 返回按钮或操作按钮组
3. 搜索/筛选区
4. 卡片或列表容器
5. 加载态
6. 空状态
7. 编辑弹窗
8. 确认弹窗

判断：

虽然业务差异比切换弹窗更大，但非常适合抽通用布局、按钮体系、弹窗骨架和状态组件。

### 2.3 样式层重复

已检查 `web/static/css/`，发现以下问题：

1. `style.css` 达到 `7235` 行，明显承担了大量全局与页面级样式。
2. `character_management.css` 达到 `5385` 行，已经接近一个独立子系统的规模。
3. `role_switcher_modal.css` 与 `player_switcher_modal.css` 高度相似。
4. 多个页面都自行定义按钮、卡片、弹窗、列表、加载态、空状态样式。

判断：

当前 CSS 已经形成“全局大文件 + 页面私有大文件并存”的状态，继续新增页面会放大维护成本。

### 2.4 脚本层重复与分叉

已检查 `web/static/js/`，发现以下问题：

1. `user_attributes.js` 与 `user_attributes_original.js` 相似度约 `0.835`。
2. `user_attributes_realtime.js` 也与现行版本存在较高相似度。
3. `user_attributes_original.js`、`user_attributes_realtime.js` 当前未发现模板引用。
4. `player_switcher_modal.js` 与 `role_switcher_modal.js` 逻辑高度相近。

判断：

脚本层已经出现“旧实现保留 + 新实现并存 + 页面只引用其中一个”的迹象，后续很容易出现修 A 漏修 B。

### 2.5 静态资源不一致

模板中发现缺失静态资源引用：

1. `web/templates/dev/hidden_settings.html` 引用了不存在的 `css/base.css`
2. `web/templates/index.html` 引用了不存在的 `css/angle_bracket_display.css`
3. `web/templates/index.html` 引用了不存在的 `js/auto_stream_handler.js`
4. `web/templates/semantic_search.html` 引用了不存在的 `css/common.css`

判断：

这类问题通常不是单点问题，而是说明前端资源管理方式缺少统一清单和清理机制。

### 2.6 大页面与大文件

体量较大的模板文件：

1. `web/templates/index.html`：`4398` 行
2. `web/templates/enhanced_summary_progress.html`：`2480` 行
3. `web/templates/image_settings.html`：`1738` 行
4. `web/templates/model_config.html`：`1562` 行
5. `web/templates/user_attributes.html`：`1415` 行

体量较大的脚本文件：

1. `web/static/js/character_management.js`：`3319` 行
2. `web/static/js/user_attributes_realtime.js`：`1706` 行
3. `web/static/js/user_attributes.js`：`1646` 行
4. `web/static/js/keyword_worldbook.js`：`1547` 行
5. `web/static/js/message_editor.js`：`1461` 行

判断：

这说明页面职责和前端模块边界还不够清晰，未来改一处牵一片的概率较高。

## 3. 优化目标

本轮优化不追求“框架升级”或“前端重写”，目标应聚焦在以下五点：

1. 降低重复模板数量。
2. 降低重复 CSS/JS 逻辑。
3. 建立统一的管理页页面壳与弹窗骨架。
4. 清理废弃资产和缺失引用。
5. 在不破坏 Flask + Jinja + 原生 JS 架构的前提下，让后续页面更容易扩展。

## 4. 优化原则

1. 以低风险、可回滚、分阶段为前提。
2. 优先抽“结构稳定但业务差异小”的部分。
3. 不先动聊天主页面 `web/templates/index.html` 的核心交互逻辑。
4. 优先从独立管理页和弹窗页收敛，再逐步处理复杂页面。
5. 每一阶段都要做到“可单独上线”，避免一次性大改。

## 5. 分阶段优化路线

### 第一阶段：建立公共骨架

目标：

先把最通用、最稳定、最容易复用的页面外壳统一起来。

建议动作：

1. 新增 Jinja 基模板，例如 `web/templates/layouts/base.html`。
2. 在基模板中统一以下内容：
   - `<!DOCTYPE html>`
   - `<html lang="zh-CN">`
   - `<meta charset>`
   - `<meta name="viewport">`
   - 公共 favicon 或基础资源位
   - 通用 CSS 插槽
   - 通用 JS 插槽
3. 为“管理类页面”新增二级布局，例如 `web/templates/layouts/management_page.html`。
4. 把通用返回按钮、页面标题区、页面容器宽度、基础间距规则统一。

预期收益：

1. 后续页面改公共资源时不再全局手改。
2. 页面骨架差异减少，前端结构更清晰。

### 第二阶段：先收敛高重复弹窗

目标：

优先处理收益最高、风险最低的一组重复页面。

建议动作：

1. 把 `role_switcher_modal` 和 `player_switcher_modal` 合并为一套可配置模板。
2. 抽一个通用切换弹窗模板，例如：
   - 标题
   - 搜索框
   - 列表容器
   - 计数区
   - footer 按钮区
3. 把 CSS 合并成一个通用文件，例如 `switcher_modal.css`。
4. 把 JS 抽成一个工厂或配置驱动模块，例如：
   - 数据加载函数
   - 渲染函数
   - 搜索字段配置
   - 点击后回调

预期收益：

1. 直接减少一组高相似度模板、脚本、样式的重复维护。
2. 后续如果再出现“场景切换器”“配置切换器”，可以直接复用。

### 第三阶段：抽管理页通用组件

目标：

把管理后台风格页面中最常出现的结构组件沉淀下来。

建议抽象的前端片段：

1. 页面头部 `page-header`
2. 顶部操作区 `page-actions`
3. 搜索栏 `search-bar`
4. 筛选栏 `filter-bar`
5. 列表容器 `list-shell`
6. 空状态 `empty-state`
7. 加载态 `loading-state`
8. 通用确认弹窗 `confirm-modal`
9. 通用编辑弹窗 `form-modal`

优先接入页面：

1. `web/templates/global_worldbook.html`
2. `web/templates/keyword_worldbook.html`
3. `web/templates/player_management.html`
4. `web/templates/event_manager.html`

暂缓页面：

1. `web/templates/index.html`
2. `web/templates/character_management.html`
3. `web/templates/user_attributes.html`

原因：

这些页面更复杂，先等通用骨架稳定后再接入，风险更低。

### 第四阶段：CSS 体系收敛

目标：

把“全局大文件 + 页面私有重复样式”逐步压缩成更清晰的层次。

建议拆分方式：

1. `base.css`
   - reset
   - 根变量
   - typography
   - body 基础规则
2. `components.css`
   - 按钮
   - 卡片
   - 弹窗
   - 表单项
   - 标签
   - 加载态
   - 空状态
3. `layout.css`
   - 容器
   - 栅格
   - header/footer 壳
4. 页面级 CSS
   - 只保留业务特有布局和视觉规则

执行要求：

1. 不要一次性拆 `style.css`。
2. 先从新抽出来的公共类开始迁移。
3. 原类名先兼容保留，等页面迁移完成后再删旧规则。

### 第五阶段：JS 模块化收敛

目标：

在不引入前端框架的前提下，把可复用逻辑收拢到共享模块。

建议优先抽离的公共能力：

1. `modal_manager.js`
2. `list_page_manager.js`
3. `search_filter_manager.js`
4. `confirm_dialog.js`
5. `toast_or_status_manager.js`
6. `file_upload_helper.js`

建议优先处理的重复逻辑：

1. 搜索输入监听
2. 列表渲染前的 loading/empty 切换
3. 新建/编辑/删除确认流程
4. 模态框打开关闭与遮罩行为
5. 批量选择、全选、反选

### 第六阶段：清理旧文件和异常引用

目标：

防止“重复实现继续长期共存”。

建议动作：

1. 确认 `user_attributes_original.js` 和 `user_attributes_realtime.js` 是否仍需保留。
2. 如果只是历史备份，迁出正式静态目录或删除。
3. 修复模板中缺失的静态资源引用。
4. 建立一个简单的静态资源巡检脚本，检查：
   - 模板引用了不存在的文件
   - 静态目录存在长期未引用文件

## 6. 建议的实施顺序

推荐按下面顺序推进：

1. 基模板 `base.html`
2. 通用管理页布局
3. 合并 `role_switcher_modal` / `player_switcher_modal`
4. 抽通用弹窗和状态组件
5. 收敛世界书类页面
6. 清理未引用脚本与缺失引用
7. 再评估是否继续拆 `index.html`、`character_management.html`

这个顺序的原因是：

1. 先做收益高、耦合低的部分。
2. 先建立复用层，再迁移复杂页面。
3. 避免一开始碰聊天主页面导致回归风险扩大。

## 7. 风险点

1. 这个项目很多功能依赖模板中的全局函数和内联 `onclick`，抽象时不能把这些入口直接隐藏掉。
2. 页面看起来相似，不代表接口、字段名、数据结构完全相同，抽公共模块时必须做配置化，而不是硬合并。
3. `index.html` 是核心页面，脚本多、依赖多，不能用“按管理页思路直接拆”。
4. 一些旧脚本可能被测试页、隐藏页、临时功能使用，删除前要全仓搜索并人工确认。

## 8. 验收标准

满足以下条件，说明优化是有效的：

1. 至少建立一套公共基模板并被多个页面使用。
2. `role_switcher_modal` 与 `player_switcher_modal` 不再各维护一整套模板/CSS/JS。
3. 通用弹窗、按钮、空状态、加载态开始集中管理。
4. 至少清理一批未引用或失效的前端静态资源。
5. 页面功能不回归，相关页面可以正常打开、搜索、编辑、保存。

## 9. 本次建议的首批落地范围

如果只做一轮“小而稳”的前端优化，我建议首批只包含下面几项：

1. 新增 `base.html`
2. 抽管理页公共头部和基础容器
3. 合并角色/玩家切换弹窗
4. 修复缺失静态资源引用
5. 清理未使用的 `user_attributes` 历史脚本

这样做的优点是：

1. 代码收益立即可见。
2. 对主聊天流程影响最小。
3. 为后续继续整理世界书页、角色页打下基础。

## 10. 结语

这个项目的前端不是“完全混乱”，而是已经进入了典型的“功能越做越多，复用层跟不上”的阶段。现在最合适的策略不是推倒重来，而是先把高重复、低风险、可配置化的部分收敛起来。只要按上面的阶段推进，后面再新增页面时，重复建设会明显下降。
